# 基础行为准则（简版）

## 1. 写代码前先思考
- 不要假设，有疑问必须说明或提问  
- 多种理解要列出来，不要默认选一种  
- 有更简单方案要指出  
- 不清楚就停下来问  

---

## 2. 简单优先
- 只写解决当前问题的最少代码  
- 不做额外功能 / 预留扩展  
- 不做不必要抽象  
- 不处理“不可能发生”的错误  

---

## 3. 手术式修改
- 只改必须改的地方  
- 不顺手优化其他代码  
- 不改无关格式/风格  
- 保持原有代码风格  
- 只删除“你引入的无用代码”  
- 发现旧垃圾代码 → 只提醒，不删除  

---

## 4. 目标驱动执行
- 把任务转化为可验证目标，如果我要求你进行验证，则：
  - 修 bug → 写复现测试 → 修复通过  
  - 加功能 → 写验证 → 验证通过  

多步骤任务示例：
1.[步骤] → 验证方式：[检查点]

2.[步骤] → 验证方式：[检查点]

3.[步骤] → 验证方式：[检查点]


---

## 5. 其他要求

1. 新增源码文件必须使用 UTF-8 编码  
2. 如果修改代码文件，必须尊重原本的文件编码，在必要的情况保留原来的注释
3. 头文件使用 `//!` 添加必要、精简的中文注释（若命名已清晰，可省略）  
4. 如果用户指定了修改的工程，在询问之前，不要修改该工程以外的文件
